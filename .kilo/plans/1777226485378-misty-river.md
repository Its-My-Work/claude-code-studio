# План вынесения переводов в отдельные файлы

## Цель
Вынесение всех переводов и локализаций из встроенного объекта `TRANSLATIONS` в `public/index.html` в отдельные JSON-файлы для каждого языка, чтобы упростить поддержку и добавление новых языков.

## Текущая структура
- Переводы хранятся в объекте `TRANSLATIONS` внутри `public/index.html`
- Поддерживаемые языки: `en` (английский), `ru` (русский)
- Функция `t(key)` возвращает перевод по ключу
- Функция `setLang(lang)` устанавливает язык и обновляет интерфейс

## Предлагаемое решение

### 1. Создание структуры файлов переводов
```
public/locales/
├── en.json    # Английские переводы
├── ru.json    # Русские переводы
└── index.js   # Загрузчик переводов
```

### 2. Формат JSON файлов
Каждый файл будет содержать объект с ключами переводов:
```json
{
  "sec.proactive": "Status",
  "sec.chats": "Chats",
  "welcome.desc": "Choose MCP and Skills on the left, files on the right.",
  ...
}
```

### 3. Модификация загрузки переводов
- Заменить встроенный объект `TRANSLATIONS` на асинхронную загрузку из файлов
- Добавить кеширование загруженных переводов
- Обеспечить fallback на английский при отсутствии файла

### 4. Изменения в коде
- Модифицировать функцию `t(key)` для работы с загруженными данными
- Модифицировать функцию `setLang(lang)` для загрузки соответствующих файлов
- Добавить обработку ошибок загрузки

### 5. Миграция существующих данных
- Экспортировать текущие переводы из `TRANSLATIONS.en` в `locales/en.json`
- Экспортировать текущие переводы из `TRANSLATIONS.ru` в `locales/ru.json`

### 6. Тестирование
- Проверить загрузку всех языков
- Проверить работу функции `t(key)`
- Проверить переключение языков через `setLang()`
- Проверить работу всех `data-i18n*` атрибутов

## Преимущества решения
- Легче добавлять новые языки (просто добавить новый JSON файл)
- Переводы отделены от кода, проще редактировать
- Возможность загрузки переводов по требованию
- Уменьшение размера основного HTML файла

## Выявленные проблемы и исправления

### Проблемы с текущей реализацией
1. **Синхронная функция `t(key)` не работает с асинхронной загрузкой**
   - Функция `t` в locales/index.js пытается загружать переводы асинхронно, но код ожидает синхронный возврат
   - Это приводит к тому, что переводы не применяются

2. **Порядок инициализации**
   - `initializeTranslations` вызывается до загрузки DOM элементов
   - Многие элементы с `data-i18n` атрибутами ещё не существуют

3. **Отсутствие fallback механизма**
   - При ошибке загрузки переводов интерфейс остаётся без текста
   - Нет перехода на старый встроенный механизм

### Исправления
1. **Сделать функцию `t` синхронной с отложенной загрузкой**
   - Функция `t` сразу возвращает ключ, но запускает асинхронную загрузку
   - После загрузки применяет переводы ко всем элементам

2. **Исправить порядок загрузки**
   - Инициализировать переводы после загрузки DOM (`DOMContentLoaded`)
   - Добавить проверку готовности DOM перед применением переводов

3. **Добавить fallback на старые переводы**
   - Если новые переводы не загрузились, использовать встроенный объект `TRANSLATIONS`
   - Обеспечить обратную совместимость

4. **Исправить логику загрузки**
   - Загружать переводы один раз при старте
   - Кешировать и переиспользовать загруженные переводы

5. **Проверить полное удаление украинских переводов**
   - Убедиться, что нет остатков украинского языка в коде
   - Проверить, что все украинские тексты заменены на русские
   - Очистить кеш браузера при тестировании
